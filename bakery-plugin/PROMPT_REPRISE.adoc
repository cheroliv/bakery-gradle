== Analyse du Bug `publishProfile`

=== 1. Symptômes observés

* Le profil GitHub `cheroliv/cheroliv` affichait le `README.adoc` du plugin `bakery-gradle` au lieu du profil personnel.
* Le `README.md` et `README_fr.md` du profil avaient disparu du repo.
* Seuls `README.adoc` et `README_fr.adoc` (doc du plugin) étaient présents.

=== 2. Cause racine

`PublishProfileTask.kt` ligne 30 :
[source,kotlin]
----
val repoDir = createRepoDir("${buildDir.absolutePath}/${pushProfile.to}", project.logger)
----
Ce répertoire est **vide** à la création.
Le `pushToRemote` dans `GitService.kt` fait ensuite :
1. `Git.init()` sur un repo vide
2. `addAllFiles` + `commitChanges`
3. `executePushCommand` avec `isForce = true`

Un force-push d'un historique à 1 commit depuis un repo vide **écrase entièrement** le repo distant — historique perdu, fichiers supprimés.

=== 3. Pourquoi les fichiers du plugin ont été poussés

Le `site.yml` de `cheroliv.com` contient :
[source,yaml]
----
profileFiles:
  - "README.md"
  - "README_fr.md"
----

Mais `pushProfile.from: ""` est vide. Dans le contexte du plugin lui-même, `projectDir` pointe vers `bakery-plugin/`, et le `copyProfileFiles` copie depuis `projectDir.resolve(fileName)`.

Si la tâche a été exécutée depuis le mauvais projet (ou que le `projectDir` résout vers le plugin au lieu du consumer), les mauvais fichiers sont copiés.

=== 4. Problèmes à corriger

* **GitService.pushToRemote** doit cloner le repo distant avant de push, ou du moins préserver l'historique existant.
* **PublishProfileTask** doit vérifier que `repoDir` contient bien les fichiers attendus avant le push.
* **publishProfile** doit utiliser un `from` explicite ou résoudre le `projectDir` correctement (doit pointer vers le consumer, pas vers le plugin).
* **SiteConfiguration** : le `pushProfile.to` est utilisé comme sous-répertoire local (ex: `profile-cvs`), mais aussi comme nom de répertoire dans `buildDir`. Confusion possible.
* **Restauration profil** : déjà faite manuellement via GH CLI — pas d'action requise.

=== 5. Règle de sûreté à ajouter

Avant tout `publishProfile` en production :
1. Vérifier `projectDir` (doit être le consumer, pas le plugin)
2. Vérifier `profileFiles` résolus en absolu
3. Vérifier que le clone/pull du repo distant est fait avant force-push
4. Option : ajouter un dry-run ou une confirmation interactive
